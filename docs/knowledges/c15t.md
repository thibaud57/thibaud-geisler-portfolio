# Description

`c15t` (prononcé "consent", Apache 2.0) est une CMP (Consent Management Platform) headless conçue pour les apps modernes React/Next.js. Elle gère la persistance du consentement, le banner et la modale de préférences, la détection de juridiction et la traduction multi-langues, avec 3 modes de déploiement : `offline` (localStorage seul, zéro backend), `hosted` (cloud c15t.dev managé) ou `custom` (self-hosted ou bring-your-own-backend).

Dans ce projet, on utilise le mode `offline` car le portfolio est single-user, n'a pas besoin d'audit trail serveur, et l'identité juridique est forcée à FR pour appliquer GDPR/CNIL inconditionnellement. Les composants `ConsentBanner` + `ConsentDialog` sont mountés dans `Providers.tsx`, leurs translations FR/EN proviennent de `@c15t/translations/all`, et le theming est fait via override des CSS variables exposées par `@c15t/nextjs/styles.css`.

# Concepts Clés

## ConsentManagerProvider en mode offline

### Description

Provider racine qui initialise le store de consentement, expose le contexte React aux descendants et configure les composants UI. Le mode `offline` stocke le consentement uniquement en localStorage + cookie, sans appel réseau ni API key. Combiné à `overrides.country: 'FR'`, il force la juridiction GDPR/CNIL sans dépendre de la géo-détection IP.

### Exemple

```tsx
'use client'
import {
  ConsentManagerProvider,
  ConsentBanner,
  ConsentDialog,
} from '@c15t/nextjs'
import { baseTranslations } from '@c15t/translations/all'

export function Providers({ children }: { children: React.ReactNode }) {
  return (
    <ConsentManagerProvider
      options={{
        mode: 'offline',
        overrides: { country: 'FR' },
        consentCategories: ['necessary', 'marketing'],
        legalLinks: {
          privacyPolicy: { href: '/fr/politique-de-confidentialite', target: '_self' },
          cookiePolicy: { href: '/fr/politique-cookies', target: '_self' },
        },
        i18n: {
          locale: 'fr',
          detectBrowserLanguage: false,
          messages: baseTranslations,
        },
      }}
    >
      {children}
      <ConsentBanner hideBranding />
      <ConsentDialog hideBranding />
    </ConsentManagerProvider>
  )
}
```

### Points Importants

- Le Provider doit être un Client Component (`'use client'`) car il utilise React Context et accède à `localStorage`
- `mode: 'offline'` n'a pas d'audit trail serveur : si un audit CNIL formel exige une preuve datée par utilisateur, basculer en mode `hosted` ou `custom` avec backend persistant
- `overrides.country: 'FR'` force GDPR sans appeler d'API géo. À retirer si l'app sert plusieurs juridictions et qu'on veut détection auto via mode hosted
- `consentCategories` : valeurs autorisées `'necessary' | 'measurement' | 'marketing' | 'experience' | 'functionality'`. La catégorie `necessary` est toujours read-only true
- `detectBrowserLanguage: false` désactive la détection auto via `navigator.language` ; sans ça, le Provider lit `navigator.language` au mount et écrase la `locale` initiale

## Composants UI : ConsentBanner & ConsentDialog

### Description

`ConsentBanner` est le bandeau bottom-left affiché tant que l'utilisateur n'a pas interagi. `ConsentDialog` est la modale de préférences granulaires (toggles par catégorie). Les noms ont changé en v2 : `CookieBanner → ConsentBanner` et `ConsentManagerDialog → ConsentDialog`. Tous deux acceptent `hideBranding` pour masquer le badge "Secured by c15t" (gratuit, fonctionne en mode offline).

### Exemple

```tsx
import { ConsentBanner, ConsentDialog } from '@c15t/nextjs'

<ConsentBanner hideBranding />
<ConsentDialog hideBranding />
```

### Points Importants

- Les 3 boutons par défaut (`Tout rejeter` / `Accepter tout` / `Personnaliser`) sont rendus au même niveau visuel = conforme exigence CNIL "rejet aussi simple que l'acceptation"
- Le banner ne réapparaît pas tant que le cookie de consentement (durée par défaut alignée GDPR ~13 mois) est valide
- Pour ouvrir la modale depuis un bouton custom (footer "Gérer mes cookies"), utiliser `useConsentManager().setActiveUI('dialog')`
- Pas de prop `position` : la position bottom-left est gérée par le CSS de la lib, modifiable via `--banner-position-*` ou `slots`
- `noStyle: true` au niveau Provider supprime tous les styles défaut (markup nu) si on veut full custom UI

## Synchronisation locale next-intl ↔ c15t

### Description

c15t et next-intl gèrent leurs i18n indépendamment. Pour que le banner switche FR ↔ EN quand l'utilisateur change de langue dans le projet, il faut appeler `setLanguage(locale)` du store c15t à chaque changement de `useLocale()` next-intl. Pattern Effet React simple.

### Exemple

```tsx
'use client'
import { useConsentManager } from '@c15t/nextjs'
import { useLocale } from 'next-intl'
import { useEffect } from 'react'

export function ConsentLanguageSync() {
  const { setLanguage } = useConsentManager()
  const locale = useLocale()

  useEffect(() => {
    setLanguage(locale)
  }, [locale, setLanguage])

  return null
}
```

À mounter à l'intérieur de `<ConsentManagerProvider>`, au-dessus de `<ConsentBanner>`.

### Points Importants

- En mode offline, `setLanguage()` ne fait pas d'appel réseau : il pioche dans le `messages` object passé au Provider (qui doit donc inclure les 2 locales)
- Ne PAS utiliser `useEffect` pour faire ce sync depuis un Server Component : le hook `useLocale()` côté Server retourne aussi la locale mais ne re-render pas le client
- Toujours inclure `setLanguage` dans les deps `useEffect` (Rule of Hooks deps exhaustives)

## Hook useConsentManager (gating de scripts tiers)

### Description

Hook principal pour lire l'état de consentement et déclencher le rendu/blocage de scripts tiers (Calendly, GA, etc.). Retourne `consents` (Record par catégorie), `has({ category })` (helper booléen), `setActiveUI` (ouvrir/fermer banner/dialog), `setLanguage`, et le state d'interaction.

### Exemple

```tsx
'use client'
import { useConsentManager } from '@c15t/nextjs'

export function CalendlyWidget() {
  const { has } = useConsentManager()
  const marketingAccepted = has({ category: 'marketing' })

  if (!marketingAccepted) {
    return <p>Acceptez les cookies marketing pour afficher le calendrier.</p>
  }

  return <iframe src="https://calendly.com/..." />
}
```

### Points Importants

- Le hook re-render le composant à chaque changement de consentement → pattern réactif natif
- `has({ category })` est préférable à `consents.marketing` car il gère les cas edge (catégorie non déclarée → retourne false)
- En mode SSR, le hook doit être utilisé dans un Client Component descendant du Provider (sinon throw)
- Pour pré-prerender un état neutre sans clignotement, gate le iframe derrière un placeholder accessible (pas un blanc)

## Theming via CSS variables (~50 vars exposées)

### Description

c15t expose un système de design tokens via CSS variables (importé via `@c15t/nextjs/styles.css`). L'override se fait dans `globals.css` après l'import, en re-définissant les vars dans `:root` et `.dark`. Variantes `-dark` natives par variable. Permet de matcher pixel-perfect les tokens DESIGN.md du projet.

### Exemple

```css
/* src/app/globals.css */
@import "@c15t/nextjs/styles.css";
@import "tailwindcss";

:root {
  --button-primary: var(--primary);
  --button-primary-hover: var(--primary);
  --button-text-primary: var(--primary-foreground);
  --button-border-radius: var(--radius);
  --button-font: var(--font-sans);
  --banner-background-color: var(--card);
  --banner-text-color: var(--foreground);
  --banner-border-color: var(--border);
  --banner-border-radius: var(--radius);
  --dialog-background-color: var(--card);
  --dialog-text-color: var(--foreground);
  --dialog-border-radius: var(--radius);
}

.dark {
  --button-primary: var(--primary);
  --banner-background-color: var(--card);
  --banner-text-color: var(--foreground);
  --banner-border-color: var(--border);
  --dialog-background-color: var(--card);
  --dialog-text-color: var(--foreground);
}
```

### Points Importants

- `@import "@c15t/nextjs/styles.css"` doit être placé AVANT `@import "tailwindcss"` pour l'ordering des layers CSS Tailwind 4
- Les variants outline/secondary ont leurs propres vars (`--button-secondary-*`, `--button-neutral-*`) à mapper séparément
- Les vars couvrent : `--button-*`, `--banner-*`, `--dialog-*`, `--accordion-*` (sections de la modale), `--card-*`. Lister toutes les vars : `grep -oE "\-\-[a-z][a-z0-9-]+" node_modules/@c15t/ui/dist/styles.css | sort -u`
- Alternative JS-side via `options.theme: { colors, radius, slots }` dans le Provider, équivalent fonctionnel mais moins lisible quand il y a beaucoup d'overrides

## Translations FR/EN via @c15t/translations/all

### Description

Le package `@c15t/translations` fournit les traductions complètes pour 30+ langues UE (dont `fr`, `en`, `de`, `es`, `it`...). L'import depuis `/all` charge l'objet `baseTranslations: Record<string, CompleteTranslations>` qu'on passe au `messages` du Provider. Permet d'éviter de retranscrire manuellement les ~20 strings du banner/dialog.

### Exemple

```tsx
import { baseTranslations } from '@c15t/translations/all'

// Toutes les locales d'un coup :
i18n: {
  locale: 'fr',
  detectBrowserLanguage: false,
  messages: baseTranslations,
}

// Override partiel d'une string spécifique :
i18n: {
  locale: 'fr',
  messages: {
    ...baseTranslations,
    fr: {
      ...baseTranslations.fr,
      common: {
        ...baseTranslations.fr?.common,
        acceptAll: 'Tout autoriser',
      },
    },
  },
}
```

### Points Importants

- `@c15t/translations` doit être installé EXPLICITEMENT (`pnpm add @c15t/translations`) même si déjà présent en transitif via `@c15t/nextjs` : sinon `Module not found: Can't resolve '@c15t/translations/all'`
- L'import `@c15t/translations` (sans `/all`) ne re-exporte que `enTranslations` : pour le multi-langue il FAUT `@c15t/translations/all`
- Override partiel : toujours spread `...baseTranslations.fr.<section>` pour ne pas perdre les autres clés du namespace
- Les noms de clés (`common.acceptAll`, `cookieBanner.title`, etc.) suivent la `CompleteTranslations` shape exposée dans les types

## Liens légaux dans le banner

### Description

`legalLinks` au niveau Provider injecte les liens "Politique de confidentialité" et "Politique cookies" dans le banner et la modale, pointant vers les pages internes du projet (sub 4 de la feature 7).

### Exemple

```tsx
options={{
  legalLinks: {
    privacyPolicy: {
      href: '/fr/politique-de-confidentialite',
      target: '_self',
    },
    cookiePolicy: {
      href: '/fr/politique-cookies',
      target: '_self',
    },
  },
}}
```

### Points Importants

- `target: '_self'` pour navigation interne (recommandé). `'_blank'` ouvre nouvel onglet (option si la page contient elle-même un appel à l'action de consentement)
- Les hrefs sont rendus tels quels : pour multi-locale, idéalement les calculer dynamiquement via `getPathname({ locale, href })` next-intl si le routing en dépend
- Pas obligatoire pour la conformité CNIL si le footer du site contient déjà ces liens accessibles, mais recommandé pour visibilité au moment du choix

# Bonnes Pratiques

## ✅ Recommandations

- **Mode offline pour ce projet** : single-user, pas d'audit trail serveur requis, zéro coût opérationnel, aucune dépendance SaaS
- **`overrides.country: 'FR'`** pour forcer GDPR/CNIL inconditionnellement et éviter une géo-détection inutile (le portfolio cible majoritairement FR/EU)
- **`hideBranding` sur Banner et Dialog** : retire "Secured by c15t" gratuitement, look 100% projet
- **Theming via CSS vars dans `globals.css`** plutôt que `options.theme` JS : plus lisible quand il y a 10+ overrides, alignement direct avec les tokens shadcn/Tailwind 4 du projet
- **Sync next-intl ↔ c15t via `<ConsentLanguageSync />`** : composant minimaliste, mounter au-dessus du banner dans Providers
- **Installer `@c15t/translations` explicitement** (pas seulement transitif) pour pouvoir importer `/all`
- **Gating des scripts tiers via `useConsentManager().has({ category: 'marketing' })`** plutôt que d'injecter le script global et l'enlever après refus

## ❌ Anti-Patterns

- Mounter `ConsentManagerProvider` dans un Server Component → throw, c'est obligatoirement un Client Component
- Utiliser les anciens noms `CookieBanner` / `ConsentManagerDialog` (v1.x) en v2 → erreur compile, exports inexistants
- Appeler `setLanguage` depuis next-intl directement : les deux i18n sont indépendants, c15t n'écoute pas next-intl. Utiliser le composant `<ConsentLanguageSync />`
- Embarquer le mode `hosted` "au cas où on en aurait besoin plus tard" → coûte un account/API key et une dépendance réseau pour aucun bénéfice tant que pas d'audit trail
- Mettre `@import "@c15t/nextjs/styles.css"` APRÈS `@import "tailwindcss"` → certains styles c15t écraseront les utilities Tailwind
- Utiliser `noStyle: true` ET essayer d'override des CSS vars en parallèle : `noStyle` retire tout, les vars n'ont plus rien à styler
- Ajouter une catégorie `analytics` dans `consentCategories` "pour préparer Umami" : Umami est cookie-less par design, n'a pas besoin de consent (cf ADR-007). YAGNI
