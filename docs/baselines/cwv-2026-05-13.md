# Baseline Core Web Vitals — 13 mai 2026

## Diagnostic

### LCP

- **Mobile encore élevé (4,1–5,4 s)** malgré la réduction bundle. Le coût de parsing JS subsiste (~1,8 s d'exécution JS restant). Probablement Framer Motion / composants Aceternity non lazy-loadés, ou image LCP sans `priority`.
- **Desktop excellent (0,8–0,9 s)** : confirme que le LCP mobile est limité par la connexion 4G lente + parsing JS, pas par le rendu serveur.

### TBT

- **Amélioration majeure** sur toutes les pages : `-580 ms` à `-1 140 ms`. Le chunk icons éliminé était le principal coupable du thread blocking.
- Reste à traiter : 460 ms TBT `/fr` mobile (seuil cible < 200 ms).

### CLS

- **CLS desktop toujours à ~0,276–0,287** (mauvais, cible < 0,1). Non impacté par l'optimisation bundle. À investiguer : image LCP du hero (dimensions figées ? `priority` ?) ou composant Magic UI / Aceternity s'insérant après hydratation.

### Accessibilité

- **96/100** (vs 100/100 en baseline). Régression à investiguer : problème de contraste signalé par Lighthouse.

## Leviers restants

1. **CLS desktop (~0,28)** : vérifier `width`/`height` sur l'image hero + composants qui s'insèrent après hydratation
2. **LCP mobile (4–5 s)** : `priority` sur l'image LCP above-the-fold, lazy-load Framer Motion / Aceternity si possible
3. **TBT `/fr` mobile (460 ms)** : profiler les tâches longues restantes (6 détectées)
4. **Accessibilité (96)** : corriger les problèmes de contraste signalés
