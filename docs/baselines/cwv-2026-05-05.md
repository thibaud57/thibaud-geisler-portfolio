# Baseline Core Web Vitals — 5 mai 2026

## Diagnostic

### LCP

- **Mobile : 16,1 s** (catastrophique). Émulation 4G lente Moto G Power, le bundle JS bloque le rendu pendant 14 s avant le contenu visible.
- **Desktop : 1,0–1,1 s** (excellent). Confirme que le LCP mobile est principalement causé par le coût de téléchargement et de parsing du JS sur connexion lente, pas par le rendu serveur.

### TBT (proxy de l'INP terrain)

- **1 040 → 1 490 ms partout**, mobile et desktop. C'est le **vrai problème**, indépendant du device. Le main thread est bloqué par l'exécution JavaScript.
- En conditions réelles, ça se traduira par un INP dégradé sur mobile (interactions saccadées au scroll/tap initial).

### CLS

- **Accueil desktop : 0,276** (mauvais). À investiguer en priorité : probablement un décalage causé par l'image LCP du portrait ou un composant Magic UI / Aceternity qui s'insère après hydratation.
- Autres pages : 0–0,059 (OK).

### Coupables identifiés par PageSpeed

| Audit | Coût |
|-------|------|
| Réduisez les ressources JavaScript inutilisées | ~2 000 Kio (≈ 2 Mo) |
| Évitez les énormes charges utiles de réseau | 3 140 Kio total (≈ 3 Mo) |
| Réduisez le travail du thread principal | 5,0 s |
| Réduisez le temps d'exécution de JavaScript | 1,9 s |
| Évitez les tâches longues dans le thread principal | 4 tâches longues détectées |
| Requêtes de blocage de l'affichage | -420 ms possibles |

## Leviers d'optimisation prioritaires

À planifier dans une feature dédiée post-MEP (hors scope checklist Post-MEP).

1. **Audit du bundle JS** (priorité 1)
   - Activer `@next/bundle-analyzer` et identifier les chunks > 100 Ko
   - Cibler les imports Magic UI, Aceternity, Framer Motion non lazy-loadés
   - Vérifier que les composants animés `client` sont bien découpés des Server Components

2. **CLS de l'accueil desktop** (priorité 2)
   - Inspecter l'image LCP (portrait dans le hero) : `width`/`height` figés ? `priority` actif ? *(cf. [`.claude/rules/nextjs/images-fonts.md`](../../.claude/rules/nextjs/images-fonts.md))*
   - Identifier tout composant qui s'insère après hydratation et déplace le layout (banner cookies, widgets)

3. **Render-blocking** (priorité 3)
   - Économies estimées 420 ms en déplaçant ou défer-ant les requêtes bloquantes (font preload, CSS critique)
