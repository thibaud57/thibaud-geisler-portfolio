# 🎯 Contexte

Le portfolio doit supporter deux langues : français (principale) et anglais. La librairie i18n doit être compatible avec Next.js App Router et gérer le routage `/fr` / `/en` via middleware.

---

# 🧩 Problème

Quelle librairie i18n utiliser pour le support FR/EN avec Next.js App Router ?

---

# 🛠️ Options Envisagées

## Option A — next-intl

**Description :** Librairie i18n conçue spécifiquement pour Next.js App Router. Support Server Components natif, middleware de routing intégré.

**Avantages :**
- Conçue pour App Router (Server Components, Server Actions — `useTranslations` et `getTranslations` disponibles dans les deux contextes)
- Middleware de routing intégré (détection langue navigateur via `Accept-Language`, redirection `/fr`/`/en`)
- Type safety complète des clés de traduction : auto-complétion et erreur TypeScript à la compilation si une clé est manquante ou mal orthographiée
- Documentation claire et active

**Inconvénients :**
- Librairie tierce à maintenir (dépendance aux releases Next.js)
- Légère courbe de prise en main (structure `messages/fr.json`, configuration middleware)

**Coût estimé :** Faible

## Option B — next-i18next

**Description :** Ancienne librairie de référence pour Next.js, basée sur i18next.

**Avantages :**
- Très mature, large écosystème i18next
- Migration facilitée depuis un projet i18next existant

**Inconvénients :**
- Conçue pour Pages Router : le support App Router nécessite des contournements non officiels (`getServerSideProps` incompatible, React Server Components non supportés nativement) — source de dette technique sur Next.js 14+
- Configuration plus complexe qu'avec next-intl

**Coût estimé :** Moyen — hacks nécessaires pour App Router, risque de régression à chaque mise à jour Next.js

## Option C — App Router built-in (routing manuel)

**Description :** Gestion manuelle des routes `/[locale]/` sans librairie dédiée.

**Avantages :**
- Zéro dépendance tierce

**Inconvénients :**
- Réinventer la roue — gestion du middleware, des dictionnaires et du routing à implémenter manuellement
- Aucune vérification à la compilation des clés de traduction manquantes ou mal orthographiées (pas de type safety)
- Maintenabilité difficile à mesure que le nombre de clés croît

**Coût estimé :** Élevé — développement initial et maintenance continue

---

# 🎉 Décision

**Option A actée — next-intl.**

Standard de facto pour Next.js App Router. Conçue nativement pour les Server Components et Server Actions, type safety des clés de traduction, middleware de routing intégré.

---

# 🔄 Conséquences

## Positives

- Type safety des clés de traduction dès le développement : auto-complétion et erreur TypeScript à la compilation si une clé est manquante
- Middleware de détection de langue automatique inclus : redirection `/fr` / `/en` via `Accept-Language` sans configuration supplémentaire
- Support FR/EN natif pour les Server Components et Server Actions

## Négatives

- Dépendance tierce à maintenir, potentiellement impactée par les breaking changes Next.js

---

# 📝 Notes complémentaires

La décision est actée. L'implémentation est prévue dès le MVP pour éviter de traduire le contenu après coup.

Structure de fichiers attendue avec next-intl :
```
messages/
├── fr.json   # langue par défaut
└── en.json
```

Langue par défaut : `/fr`. Détection via header `Accept-Language` dans le middleware Next.js, redirection 307 vers `/fr` ou `/en`.

## Sous-question ouverte : stratégie des slugs

Quelle que soit la librairie retenue, une décision complémentaire reste à prendre sur la stratégie des slugs dans les routes dynamiques :

**Option S1 — Slug language-agnostic (recommandée)** : un seul slug par entité en BDD (ex : `mon-projet`), identique dans toutes les locales. Les URLs deviennent `/fr/projets/mon-projet` et `/en/projets/mon-projet` — seul le préfixe de locale change.

- Avantage : simplicité BDD (une colonne `slug`), pas de synchronisation entre slugs
- Avantage : `generateStaticParams` trivial (une seule valeur à générer par projet)
- Inconvénient : le slug peut sembler "français" depuis une URL anglaise

**Option S2 — Slug localisé** : un slug par locale en BDD (ex : `mon-projet` en FR, `my-project` en EN). URLs : `/fr/projets/mon-projet` et `/en/projects/my-project`.

- Avantage : URL 100% cohérente avec la langue
- Inconvénient : deux colonnes slug en BDD, synchronisation à gérer, `generateStaticParams` plus complexe (un passage par locale)

**Question complémentaire : préfixes de route traduits ?** Indépendamment du slug, les segments de chemin peuvent être traduits (`/fr/projets/` vs `/en/projects/`) ou unifiés (`/fr/projects/` et `/en/projects/`). À décider avec next-intl (support via `pathnames` dans la config).

Cette sous-question sera tranchée au moment de l'implémentation de l'i18n.

## Sous-décision : content bilingue côté DB

**Options envisagées :**

- **A. Colonnes jumelées `*Fr`/`*En`** sur chaque modèle contenant du texte éditorial (`Project.titleFr`/`titleEn`, `descriptionFr`/`descriptionEn`, `caseStudyMarkdownFr`/`caseStudyMarkdownEn`, `Tag.nameFr`/`nameEn`). Helper pur `localizeProject(project, locale)` / `localizeTag(tag, locale)` côté query qui pick le champ selon la locale.
- **B. Table `Translation` normalisée** (`entityId`, `entityType`, `locale`, `field`, `value`). Scalable pour N locales, plus complexe côté requêtes.
- **C. JSON multilingue** dans une seule colonne (`title: { fr, en }`) en JSONB Postgres.

**Décision : A.**

**Rationale :**
- Portfolio single-user visant 2 locales (FR, EN) stables, pas de roadmap > 5 locales : surcoût structurel de (B) non justifié.
- Queries simples (pas de JOIN translation), types Prisma auto-générés cleanly, IDE autocomplete sur `titleFr` / `titleEn`.
- Le helper `localizeProject` isole l'implémentation : migration post-MVP vers (B) possible sans casser l'API publique (queries continuent de retourner `LocalizedProjectWithRelations` avec `title` résolu).
- (C) rejetée : JSONB → typage faible en bout de chaîne, Zod validation runtime nécessaire pour chaque lecture.

**Convention mixte :**
- **Content éditorial long** (titres, descriptions, markdown, noms de tags) → colonnes jumelées `*Fr`/`*En` sur Prisma.
- **UI chrome et labels bornés** (nav, boutons, labels d'enum comme `CompanySector.ASSURANCE`) → `messages/{fr,en}.json` consommés via `useTranslations` / `getTranslations`.
- **`Company.name`** reste non-traduit (marque commerciale conservée telle quelle, ex : "Foyer Group", "PaySystem").

**Paramètre `locale` obligatoire sur les queries :**
- `findManyPublished({ type?, locale })` et `findPublishedBySlug(slug, locale)` exigent la locale sans valeur par défaut. TypeScript force le caller à passer explicitement la locale de la route (obtenue depuis `setupLocalePage(params)`). Un oubli = erreur de compilation, pas un bug silencieux FR-only en prod.
